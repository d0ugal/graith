# Todo List as a First-Class Feature

A shared, persistent, queryable todo list managed by the daemon — a first-class
graith primitive alongside the store, messaging, and scenarios. Items survive
session end, are visible across a session subtree (parent and children) or a
whole scenario, and can be **claimed atomically** so parallel agents pulling
from one list never double-work the same item. Agents drive it through
`gr todo …` and a matching MCP tool; the human sees "what's left" across the
fleet in the overlay and the GUIs.

## Background

An AI agent's plan today lives in its context window. Claude Code renders a
todo checklist while it works, but that list is ephemeral — it is a byproduct
of the model's turn, not a durable object graith can see. When the session
stops, resumes into a fresh context, compacts, or hands off to a sibling, the
plan is gone. This is the same fragility the
[scenarios design](2026-06-22-scenarios.md) calls out: "the setup lives in the
AI's context window and is lost when the session ends."

graith already owns several coordination primitives the agent and the human
share, and each is instructive for how a todo list should be built:

- **The document store** (`internal/store/store.go`,
  [design](2026-06-16-store-refactor-design.md)) — a flat-file, git-backed
  key/value store, per-repo plus a daemon-wide `shared` namespace
  (`SharedStorePath`, `store.go:149`). Operations
  (`Put`/`Get`/`List`/`Append`/`Remove`) are plain filesystem writes serialized
  by a `store.lock` flock and committed to git for history. It is deliberately
  schema-less: keys are paths, bodies are opaque bytes. The `store` CLI path
  touches the filesystem directly with no daemon connection (the GUI reaches it
  over control messages).
- **Inter-agent messaging** (`internal/daemon/msgstore.go`) — a SQLite-backed
  (`messages.sqlite`, WAL) pub/sub + inbox system. `Publish` inserts a message
  and fans it out to live in-process subscribers (`s.subs[stream]`); `Read`,
  `Ack`, and cursors track per-subscriber unread state. The daemon publishes
  its own automated notices under a `graith:system` sender (`isSystemSender`).
- **Scenarios** (`internal/daemon/scenario.go`,
  [design](2026-06-22-scenarios.md)) — declarative multi-session fleets. Each
  scenario session carries a `role`/`task`, and today `gr scenario task-done`
  (`ScenarioTaskDone`, `scenario.go:860`) flips a single `TaskDone bool` on the
  daemon-side `ScenarioSession` state struct (`scenario.go:890`), reflected onto
  the wire `ScenarioSessionInfo` (`messages.go:894`). It records that the
  *calling member* declares its own assigned task complete — one boolean per
  session, no sub-items, no ordering, no claiming. **This design removes it and
  replaces it with the todo system** (see the scenario-integration section); it
  is the closest existing "tracked work" primitive and its limitations are
  exactly what motivate the replacement.
- **The session tree** — every `SessionState` has a `ParentID`
  (`state.go:56`) and an optional `ScenarioID` (`scenario.go`). `isDescendantOf`
  (`auth.go:335`) walks the parent chain, and this is how hierarchical
  operations (`gr stop --children`, `gr msg send --parent`) and scenario
  authorization already scope themselves.

The wire protocol is a `{"type":…,"payload":…}` envelope on channel `0x00`
(`protocol/messages.go`, dispatched in `daemon/handler.go`). Every control
message is classified in `remoteMessagePolicy` (`daemon/authmatrix.go`) — a
missing entry fails closed and trips `TestRemoteMatrixCompleteness`. That table
is **Gate A for *remote* (tailnet) connections only**: a local Unix-socket
caller bypasses it and is governed by the handler's own authorization checks
(`authmatrix.go:3`). Every wire struct is also reflected into the Go↔Swift
conformance manifest (`protocol/manifest.go`,
[protocol-conformance-design](2026-07-14-protocol-conformance-design.md)).

Agents also reach the daemon over MCP: `internal/mcp/server.go` registers tools
with `gomcp.AddTool` (`list_sessions`, `create_session`, `publish_message`,
`read_inbox`, …). Each handler is a thin proxy — it dials the daemon socket via
`s.connect()` and sends the same control message the CLI would
(`createSession`, `server.go:255`). So a CLI verb and an MCP tool are two front
doors onto one daemon operation.

The GUIs (iOS + macOS) bind to a shared Swift layer, `GraithSessionKit`
([shared-session-feature-layer](2026-07-14-shared-session-feature-layer.md)),
and every frontend capability is tracked in the hand-maintained matrix at
`internal/capabilities/capabilities.json`, guarded against doc- and code-drift.

## Problem

1. **Plans do not survive the context window.** A stopped, resumed, compacted,
   or handed-off agent loses its plan. There is no durable record of "what I was
   going to do" that graith — or a successor agent — can read back.

2. **No shared view across a subtree or fleet.** A parent orchestrator and its
   children, or the sessions in a scenario, have no common list of outstanding
   work. Coordination is improvised over free-text messages, which are a log,
   not a queryable task set.

3. **No safe hand-out for parallel agents.** When an orchestrator wants three
   agents to drain a backlog, there is no collision-free way to hand out items.
   Two agents reading the same list will both "start" the same task — a
   read-then-write TOCTOU race. The issue's competitive analysis
   ([amux/mixpeek](https://github.com/d0ugal/graith/issues/591#issuecomment-4840251519))
   identifies **atomic claim** as the load-bearing correctness primitive.

4. **The human has no "what's left" at a glance.** `gr list` shows session
   status, not the work each session is carrying. There is no fleet-wide answer
   to "how much is outstanding, and where is it blocked?"

5. **`task-done` is too coarse.** A scenario session's work is a single
   boolean. It cannot express "3 of 7 items done, 1 blocked" — the granularity
   an orchestrator actually needs to reason about progress.

## Goals

- A **durable** todo list owned by the daemon, surviving session end, resume,
  compaction, and daemon restart.
- **Item shape** rich enough to be useful: id, title, status
  (`todo`/`in-progress`/`done`/`blocked`), owner, optional parent item, tags,
  timestamps.
- **Scoping** to a session subtree by default, with the option to scope to a
  scenario — matching how graith already draws coordination boundaries.
- **Atomic, collision-free claim**: a compare-and-set `todo → in-progress`
  bound to a session, so parallel agents never double-claim (issue's
  load-bearing requirement).
- **Two front doors**: `gr todo …` CLI and a matching MCP tool, so agents
  manage items directly and the human drives from the shell.
- **Surfacing**: a fleet-wide and per-session view in the overlay, and a
  read (at least) view in the GUIs via `GraithSessionKit`.
- **Optional events**: state changes can emit pub/sub so triggers (#592, #593)
  and reviewers can react.
- **Replace scenario `task-done`**: scenarios express per-member progress
  through assigned todo items, not a single boolean — the coarse primitive is
  removed, not left alongside (breaking change, accepted).
- Clear-eyed on **overlap**: complement approvals (they stay), *replace*
  `task-done`.

### Non-Goals

These sit outside the plan because of a real external dependency or boundary —
not because they are being postponed. (Everything the plan itself touches is
committed across Phases 1–8; nothing in it is deferred.)

- **A visual kanban / drag-drop board.** The issue floats a vibe-kanban-style
  board ([comment](https://github.com/d0ugal/graith/issues/591#issuecomment-4842556611)),
  but a board is a *web presentation* on top of this data model and depends on
  the #628 web surface, which does not exist yet. This design deliberately makes
  the data model board-ready (status columns, one-level sub-items, tags,
  position ordering) so #628 can render a board without a schema change — but the
  board itself lands with #628, not here.
- **A full task-dependency graph.** Sub-items (one level of parent/child) are in
  scope; arbitrary DAG dependencies and topological scheduling are #641's
  problem, not this doc's.
- **Cross-machine todo lists.** Like scenarios, todos are single-daemon — an
  architectural boundary, not a phase.
- **Replacing approvals.** They stay — a deliberate design decision (see the
  overlap section), because approvals are a blocking gate and todos are
  non-blocking intent. (`task-done` is *not* in this list: Phase 7 removes and
  replaces it.)
- **Rich text / attachments / comments on items.** Titles + tags + an optional
  one-line note. Discussion happens in messaging, which items can reference.

## Proposals

### Proposal 0: Do Nothing

Agents keep planning in-context; scenarios keep using the one-bit `task-done`;
orchestrators keep improvising over messages and the store.

This is the status quo, and it is genuinely insufficient for the reasons in the
Problem section: no durability, no shared subtree/fleet view, and — most
sharply — **no collision-free hand-out**. To be precise about the unsafe case:
statically-assigned scenario work (each session owns its task up front) is
*not* made unsafe by the absence of todos. What is unsafe is **competitive
workers pulling from a shared backlog** without an atomic allocator — the moment
two agents race to "take the next item", the read-then-write loses one's work.
That specific pattern is what graith cannot do safely today. A motivated user
*can* approximate a todo list today by
`gr store append`-ing JSONL lines, but the store has no atomic CAS (it is a
whole-file flock, not a row-level compare-and-set), no schema, no query, and no
surfacing. Rejected.

### Proposal 1: Daemon-owned todo subsystem with atomic claim (Recommended)

A purpose-built todo subsystem in the daemon, backed by SQLite, exposed through
control messages, a `gr todo` CLI, and an MCP tool, scoped to a session subtree
or a scenario, with an atomic compare-and-set claim as its correctness
centrepiece.

#### Data model

One `todos` table in a new SQLite database, `todos.sqlite`, in the data dir — a
new `TodosDB` field on the `config.Paths` struct (`paths.go:29`, alongside
`MessagesDB`) and its assignment in `ResolvePaths` (`paths.go:88`) +
`WithDataDir` (`paths.go:109`). SQLite — not the flat-file store — is the right
substrate: the workload is high-churn row-level state transitions with indexed
queries and an atomic per-row compare-and-set, none of which the git-backed
store fits (see Proposal 2). We already run SQLite for messaging, so this adds
no dependency.

**A separate DB, not a `todos` table in `messages.sqlite`.** Note the jailstore
is *not* a standalone subsystem — `Jail`/`MarkReleased`/… are methods on
`MsgStore` and `jailed_comments` lives in `messages.sqlite`
(`jailstore.go`, sharing the `MsgStore.mu` + connection). Todos take the
opposite lifecycle choice deliberately: an independent DB isolates a
higher-churn write stream from the message log, lets todo retention/backup
evolve independently, and keeps the two mutexes uncoupled. The cost is real and
must be accounted for — its own open/close, WAL sidecar files, `--data-dir`
override, `gr doctor` visibility, and migration path — but the isolation is
worth it. (If contention or operational simplicity later argues otherwise,
folding it into `messages.sqlite` is a mechanical change.)

**The DSN carries the correctness contract.** The database must be opened with
the same pragma string `msgstore.go:51` uses —
`?_pragma=journal_mode(wal)&_pragma=busy_timeout(5000)`. The `busy_timeout` is
load-bearing, not incidental: without it a second concurrent writer can get
`SQLITE_BUSY` and `Claim` returns `(false, err)` — an *error*, not the clean
"already claimed, `n==0`" the claim contract below promises. The outcome is
still safe (no double-claim) but the *return contract* depends on the pragma.

```go
// internal/daemon/todostore.go
type TodoItem struct {
    ID        string   `json:"id"`         // "td-" + random hex
    Title     string   `json:"title"`
    Status    string   `json:"status"`     // todo | in-progress | done | blocked
    Scope     string   `json:"scope"`      // "session:<root-id>" | "scenario:<id>"
    Owner     string   `json:"owner,omitempty"`   // claimant session ID (server-derived)
    Assignee  string   `json:"assignee,omitempty"`  // responsible member (scenario completion; replaces task-done)
    ParentID  string   `json:"parent_id,omitempty"` // one-level sub-item
    Tags      []string `json:"tags,omitempty"`
    Note      string   `json:"note,omitempty"`      // optional one-liner
    CreatedBy string   `json:"created_by"`          // session ID / "human"
    CreatedAt string   `json:"created_at"`
    UpdatedAt string   `json:"updated_at"`
    Revision  int64    `json:"revision"`            // bumped on every mutation (see Events)
    Position  int64    `json:"position"`            // manual ordering within a scope
}
```

Schema (mirroring the `initSchema` style in `msgstore.go:67`), with `tags` in a
child `todo_tags` join table so tag filters are indexable, and `CHECK`
constraints enforcing the state machine at the storage layer:

```sql
CREATE TABLE IF NOT EXISTS todos (
    id          TEXT PRIMARY KEY,
    title       TEXT NOT NULL CHECK (length(title) <= 500),
    status      TEXT NOT NULL DEFAULT 'todo'
                CHECK (status IN ('todo','in-progress','done','blocked')),
    scope       TEXT NOT NULL,
    owner       TEXT NOT NULL DEFAULT '',
    assignee    TEXT NOT NULL DEFAULT '',
    parent_id   TEXT REFERENCES todos(id) ON DELETE CASCADE,
    note        TEXT NOT NULL DEFAULT '' CHECK (length(note) <= 2000),
    created_by  TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL,
    revision    INTEGER NOT NULL DEFAULT 1,
    position    INTEGER NOT NULL DEFAULT 0
    -- an in-progress/done/blocked item must have an owner; a todo must not
    CHECK ((status = 'todo' AND owner = '') OR (status <> 'todo' AND owner <> ''))
);
CREATE INDEX IF NOT EXISTS idx_todos_scope  ON todos(scope, status, position, id);
CREATE INDEX IF NOT EXISTS idx_todos_owner  ON todos(owner) WHERE owner <> '';
CREATE INDEX IF NOT EXISTS idx_todos_parent ON todos(parent_id) WHERE parent_id IS NOT NULL;
```

Parentage is enforced one-level (a `parent_id` whose own `parent_id` is non-null
is rejected at insert) and same-scope (a sub-item shares its parent's scope);
`ON DELETE CASCADE` removes sub-items with their parent.

**Scope is the primary key of "which list," and it is a *stable anchor* — not
the creating session's own id.** This is the load-bearing scoping decision. A
`session:<root-id>` scope names the **root of a session subtree**: on
`todo add`, the daemon resolves `GRAITH_SESSION_ID` **up its `ParentID` chain to
the topmost non-system ancestor** and anchors the item there, so a parent `P`
and its child `C` both write `session:P` and genuinely share one list. (An
explicit `--session <id>` or `--subtree-root` overrides the auto-anchor for the
rare case an agent wants a sub-list at itself.) A `scenario:<id>` scope is
resolved from the authoritative `ScenarioState` — membership is
`ScenarioState.SessionIDs` (which includes shared sessions), not merely a
caller's `SessionState.ScenarioID`, and the "scenario orchestrator" override is
`ScenarioState.OrchestratorID`. There is no free-floating global list — every
item is anchored, the way every session has a `ParentID`/`ScenarioID`.

Because scope is a single anchor string, both `List(scope)` and
`ClaimNext(scope)` operate on one key with no union query — the fan-out drain
works directly. **Membership**, for read/create/claim, is then a directional
predicate over the live tree: a caller is in a `session:<root>` scope iff
`caller == root || isDescendantOf(caller, root)` (`auth.go:335`), evaluated
under `sm.mu`. See the ownership section for the full predicate set, and the
lifecycle open question for what happens when the anchor root is purged or a
session is reparented.

#### The claim: atomic compare-and-set

The load-bearing detail from the issue. Claiming is a single conditional
`UPDATE` whose `WHERE` guards the pre-state, in the same spirit as the
jailstore's conditional release (`jailstore.go:178`,
`UPDATE … WHERE id = ? AND released_at = ''`) — though note the jailstore also
holds `MsgStore.mu` around its read-then-update and ignores `RowsAffected`; the
todo CAS relies on the SQL alone and *is* the stronger pattern:

```go
// owner is NEVER caller-supplied — it is ac.sessionID, derived server-side
// from the authContext, exactly as msg publishing overwrites sender identity
// (handler.go). A session cannot claim on another session's behalf.
func (s *TodoStore) Claim(id, owner string) (bool, error) {
    res, err := s.db.Exec(
        `UPDATE todos SET status='in-progress', owner=?, revision=revision+1, updated_at=?
         WHERE id=? AND status='todo' AND owner=''`,
        owner, now, id)
    if err != nil { return false, err }        // e.g. SQLITE_BUSY — surfaced, not swallowed
    n, err := res.RowsAffected()               // error checked, not discarded
    if err != nil { return false, err }
    return n == 1, nil   // exactly one claimant wins; losers get n==0
}
```

Two agents racing to claim the same item: SQLite serializes the writes, the
first `UPDATE` matches and sets `owner`, the second re-evaluates its `WHERE`
against the already-mutated row and matches zero. The loser is told "already
claimed" and moves on. No TOCTOU window, no read-then-write, no Go mutex needed
for correctness.

`ClaimNext(scope, owner)` — the fan-out drain — must be **one** statement too,
so the subquery cannot select a row another claimant already took. It targets
the lowest-`position` unclaimed item, with `id` as a deterministic tie-breaker
(`position` is not unique), and uses `RETURNING` to hand back the exact row it
claimed (a follow-up `SELECT` is *not* part of the CAS and could observe a later
state):

```sql
UPDATE todos SET status='in-progress', owner=?, revision=revision+1, updated_at=?
WHERE id = (
    SELECT id FROM todos
    WHERE scope=? AND status='todo' AND owner=''
    ORDER BY position ASC, id ASC LIMIT 1
)
RETURNING id, title, status, owner, position;
```

The regression test drives N goroutines through `ClaimNext` over N items and
asserts each item id is returned exactly once (not merely N claimers against one
explicit id).

#### The transition table (why generic update can't bypass the claim)

A single `todo_update` that accepts an arbitrary `status` would let a client set
`status='in-progress'` **without** going through the claim CAS — making
"parallel agents never double-claim" true only for well-behaved clients. So the
generic `todo_update` may **only** edit title/tags/note/position; it may never
mutate `scope` or `owner`, and every *status* change is a distinct, guarded
operation with its pre-state in the `WHERE`:

| Operation | Guarded transition (conditional `UPDATE`) |
|---|---|
| claim / start | `status='todo' AND owner='' → in-progress, owner=caller` |
| done | `owner=caller (or override) AND status='in-progress' → done` |
| block | `owner=caller (or override) AND status='in-progress' → blocked` |
| reopen | `status IN ('in-progress','blocked','done') → todo, owner=''` (owner/override) |

Because `owner` is server-derived and `scope`/`owner` are immutable through the
generic path, there is no authorization/state TOCTOU: even if the handler reads
the item to authorize before updating, the conditional predicates in the
`UPDATE` are what actually commit the transition.

#### Ownership and mutation rules

Answering the issue's open question directly. Let *caller* = `ac.sessionID`,
*root* = the item's subtree-anchor (or, for a scenario scope, its member set /
`OrchestratorID`). The predicates, all evaluated under `sm.mu`:

- **Read / create**: caller is **in scope** — for a session scope,
  `caller == root || isDescendantOf(caller, root)` (`auth.go:335`); for a
  scenario scope, `caller ∈ ScenarioState.SessionIDs`. The human (local CLI) is
  in every scope.
- **Claim**: any **in-scope** session may claim an unclaimed item; `owner` is
  set to the caller server-side (never trusted from the payload).
- **Transition a claimed item** (done/block/reopen/edit/reorder/delete): the
  **owner**, an **override authority**, or the **human**. The override authority
  is the subtree anchor root (for a session scope) or `ScenarioState.OrchestratorID`
  (for a scenario scope). A plain sibling may not steal another agent's
  in-progress item.
- **The human always wins** (`roleLocalHuman` / paired human), consistent with
  every other subsystem.

Two precision notes the reviews surfaced. First, the closest existing precedent
is `checkScenarioOp` (`auth.go:244`), **not** a `checkScenarioManage` — and it
permits the orchestrator *and any descendant* to manage a scenario. The todo
override rule above is deliberately **stricter** (owner / anchor-root /
`OrchestratorID` / human — not every descendant), because a peer agent draining
the same backlog should not be able to close a sibling's item. It is
*informed by* `checkScenarioOp`/`checkTarget`/`isDescendantOf`, not identical to
them. Second, **the auth-matrix row is not the authorization** — it only gates
what a *remote* caller may reach (Gate A). The real scope/ownership enforcement
is a new `checkTodoMutate(ac, item)` helper alongside the other `check*`
functions in `auth.go`, applied to every caller (local and remote), so the
subsystem inherits the fail-closed local-auth model unchanged.

**Human claim semantics.** `todo_claim` is a *session-grabs-work* primitive, so
the default is that the **human does not claim** — the human *assigns* (creates
+ can transition any item) but leaves `owner` to the session that picks the
item up. If manual triage later wants a human owner, the sentinel is
`owner = "human"` (mirroring `created_by`), decided explicitly rather than left
undefined.

**Stranded claims (reclaim).** A worker can claim an item and then stop or crash
without finishing it, leaving it `in-progress` owned by a dead session — the
operationally dangerous case for the parallel-drain story. Two defences: (1) the
daemon **auto-reopens** items owned by a session that transitions to
stopped/soft-deleted (a conditional `UPDATE … SET status='todo', owner=''
WHERE owner=? AND status='in-progress'`, run from the same stop path that clears
other per-session runtime state); and (2) `reopen` is available to the
override authority and human for manual recovery. A time-based claim **lease**
(auto-reopen an `in-progress` item after `[todo] claim_lease` of no progress) is
built alongside these in Phase 1; only its default *value* is left open for
review.

#### Storage: separate subsystem, not a store view

Answering the second open question: **a separate subsystem**, not a typed view
over `gr store`. The store is the wrong substrate for three concrete reasons —
(1) a row-level atomic CAS needs a specialised locked-transaction API the store
does not have (it *could* be built, but see Proposal 2), (2) a git commit per
status flip is far too heavy for the write rate a live todo list sees, and (3)
queries ("all blocked items in this scenario", "next unclaimed") want an indexed
table, not a directory walk. The todo subsystem is its own SQLite database
(`todos.sqlite`) — deliberately *not* folded into `messages.sqlite` the way the
jailstore's tables are (see the data-model section for that lifecycle choice).
(A read-only `gr todo export` — dumping a scope to a markdown/JSON doc in the
store for archival — is committed as Phase 7, but the source of truth is always
the table.)

#### Control protocol

New **request and response** message types in `protocol/messages.go` — both
directions, not just the requests: a `TodoItemInfo` wire item, a
`TodoListResponse`, and claim/mutation responses, so the GUI and CLI bind to a
stable, manifest-guarded contract rather than the daemon-package `TodoItem`
serialized ad hoc (which would evade the conformance manifest entirely). Each
exported struct is registered in `registeredTypes` + classified in
`swiftAnnotations` and the manifest regenerated
(`go test ./internal/protocol -run TestManifestUpToDate -update`); the
GUI-required item/response types are marked `required`, not deferred `planned`.
Each *message* also needs a `remoteMessagePolicy` row (`authmatrix.go`) or
`TestRemoteMatrixCompleteness` fails:

| Message | Payload | Policy | Notes |
|---|---|---|---|
| `todo_add` | scope, title, tags, parent_id, note | `remoteHumanRW` | create |
| `todo_list` | scope, filter (status/tag/owner) → `TodoListResponse` | `remoteHumanRW` | full item bodies (see below) |
| `todo_claim` | id **or** scope (claim-next) → claim response | `remoteSessionOnly` | atomic CAS; owner server-derived |
| `todo_update` | id, title/tags/note/position | `remoteHumanRW` | never scope/owner; status has own ops |
| `todo_done` / `todo_block` / `todo_reopen` | id (+ note for block) | `remoteHumanRW` | guarded status transitions |
| `todo_remove` | id | `remoteHumanRW` | cascades sub-items |

`todo_claim` is `remoteSessionOnly` — a session acting on itself, like
`approval_request`/`status_report` (`authmatrix.go:60`). The intent is precise:
claiming is a *session-initiated* action, so a remote paired human cannot claim
(a local human can, since local connections bypass Gate A). This is not about
impersonation per se; it is that "grab work" is something a *session* does.

**Guest visibility.** `todo_list` is `remoteHumanRW`, **not** `remoteReadOnly` —
todo titles/notes are free text that may name sensitive work ("fix the auth
bypass in `login.go`"), exactly the reasoning that makes store bodies
`remoteHumanRW` (`authmatrix.go:89`) and withholds jailed-comment bodies from
non-authorized callers (`mayReadJailBody`). A read-only remote guest therefore
sees only **redacted counts** surfaced through the guest-visible `SessionInfo`
(the `done/total` column), never item bodies. Handler cases go in
`daemon/handler.go`'s dispatch loop, each delegating to a `SessionManager`
method. Because scope resolution reads the session tree, the handler holds
`sm.mu` and calls into the `TodoStore` (which owns its own `db` + `mu`) — a
documented two-lock order (`sm.mu` first, then the store's), so lock ordering is
explicit and deadlock-free; the `TodoStore` methods do **not** take `sm.mu`
themselves.

#### CLI surface

`internal/cli/todo.go`, one file, mirroring `cli/scenario.go`'s shape:

```bash
gr todo add "Wire the claim CAS" --tag backend --tag p1   # add to my scope
gr todo add "Sub-task" --parent td-abc123                 # one-level sub-item
gr todo list                          # my subtree's items, grouped by status
gr todo list --scenario tracing       # a scenario's shared list
gr todo list --status blocked --all   # fleet-wide, filtered (human)
gr todo claim td-abc123               # atomic claim → in-progress, owned by me
gr todo next                          # claim the next unclaimed item in my scope
gr todo start td-abc123               # alias: claim
gr todo done td-abc123                # → done
gr todo block td-abc123 "waiting on API review"  # → blocked + note
gr todo reopen td-abc123              # → todo, clears owner
gr todo rm td-abc123
```

Scope auto-resolves from `GRAITH_SESSION_ID` **anchored to the subtree root**
(the daemon walks the `ParentID` chain, per the data-model section), so an agent
and its parent share one list; `--scenario <name>` and `--session <id>`
override; `--all` (human/orchestrator) spans scopes. Agent mode auto-enables
`--json` (the existing `GR_AGENT_MODE` detection), so an agent gets structured
output for free.

#### MCP tool surface

New tools in `internal/mcp/server.go` via `gomcp.AddTool`, each a thin proxy
over the control messages above (the `createSession` pattern, `server.go:255`):

- `todo_list(scope?, status?, tag?)` → items
- `todo_add(title, tags?, parent_id?, note?)` → item
- `todo_claim(id?)` → `{claimed: bool, item}` (id omitted = claim next; owner is
  the calling session, server-derived)
- `todo_update(id, title?, tags?, note?)` → item (never status/scope/owner)
- `todo_done(id)` / `todo_block(id, note)` / `todo_reopen(id)` → the guarded
  status transitions

An agent can then plan durably in one call and drain a shared backlog with
`todo_claim` — the fan-out story the issue is really about.

#### Events (optional, on by default for scenario scopes)

Answering the events open question: **yes, opt-in pub/sub on state change**,
reusing the msgstore. On every mutation the daemon publishes a compact JSON
event to a well-known topic — `todo:<scope>` — via `MsgStore.Publish` with the
`graith:system` sender (so it is flagged `System`, `msgstore.go:219`):

```json
{"event":"claimed","id":"td-abc","scope":"scenario:sc-1","owner":"def456","status":"in-progress"}
```

A **subscribed** session — e.g. a reviewer running
`gr msg sub --topic todo:scenario:sc-1 --follow` — reacts to "an item went
`blocked`" without polling. The reviews correctly noted that **today's trigger
sources are only `schedule` and `watch`** ([triggers-design](2026-07-11-triggers-design.md)),
so a trigger cannot consume a topic yet. Rather than assert a capability that
does not exist, this design **closes that gap itself**: Phase 5 adds a third
`[trigger.message]` source (topic-glob subscription with its own binding/restart
semantics), so a trigger firing on "todo blocked" is committed work here, not a
hoped-for follow-up. A GUI still does not receive events for free; it opens a
subscription (or polls the topic) — wired in Phase 6.

Delivery is **best-effort and fail-open** — the table is the source of truth.
Because todo and message data live in separate SQLite DBs, a mutation and its
event cannot be one transaction, and concurrent mutations can publish
out-of-commit-order. That is why every item carries a `revision`: a consumer
treats an event as a hint and **re-reads the row**, using `revision` to discard
a stale event. The "exactly one event per mutation" test is therefore qualified:
one event on the *success* path; a publish failure is logged, not fatal.

Emission is configured by a **tri-state** `[todo] emit_events` — `"scenario"`
(default: emit for scenario scopes only, to keep lone-session noise down),
`"all"`, or `"off"` — since a single boolean cannot express the intended
"on for scenarios, off for lone sessions" default.

Note the distinction the notify machinery already draws (`daemon/notify.go`): a
bare `MsgStore.Publish` to a topic reaches *subscribers and the GUI* but does
**not** wake a stopped agent — only `notifyFromDaemon` (`notify.go:79`) routes an
event into `inbox:<id>` and auto-resumes the session. So a todo event uses plain
`Publish` by default (observation), and *escalates* to `notifyFromDaemon` only
for the cases worth interrupting an agent over — e.g. "an item you own was
`blocked`" delivered to that owner's inbox, matching how PR/CI notices already
wake their owning session.

#### Surfacing in the TUI and GUI

The reviews were sharp here: a *count* and a *view of items* need different
mechanisms, and only the count rides the existing session-metadata path.

- **The count column (cheap, existing machinery).** Both `gr list` and the
  overlay picker draw trailing columns from one source of truth,
  `SessionColumn` in `internal/client/sessioncols.go` (`ShowCLI`/`ShowTUI` flags,
  paired `CLIValue`/`TUIValue` formatters). A `done/total` column added there
  appears in *both* surfaces — the overlay delegate iterates `tuiColumns()`
  (rendered at `overlay.go:681`) and the panel width grows in `totalWidth`
  (`overlay.go:210`, fed by `computeColumnWidths` at `:517`). The count reaches
  the client via the derive-and-publish path
  ([sidebar-metadata-design](2026-06-25-sidebar-metadata-design.md)): a runtime
  `json:"-"` field on `SessionState`, refreshed by the daemon after every todo
  mutation (and rebuilt on restart, since it is not persisted), copied in
  `toSessionInfo` onto `protocol.SessionInfo`. **Attribution is defined**: the
  column shows the count for the session's *own* anchored subtree list; a
  scenario-wide total is shown only on the scenario's orchestrator session (not
  echoed onto every member, which would inflate fleet totals).
- **A real item view is not a `sessionsForView()` filter.** `sessionsForView()`
  returns `[]protocol.SessionInfo` and the delegate only renders
  `sessionItem`/`groupHeader` (`overlay.go:584`) — that can show *sessions that
  have* todos, not todo rows. A fleet-wide **"Todos"** view listing individual
  items therefore needs its own thin machinery: a `todo_list` fetch (async, like
  `fetchPreview`), a `todoItem` list-model + delegate, and refresh on the
  `todo:<scope>` event (or a periodic poll). It is added as a new
  `viewMode` member (the enum `viewAll`…`viewDeleted` starts at `overlay.go:37`)
  but backed by a todo query, not the session feed. The per-session preview
  panel (`overlay.go:2226-2321`) similarly gets item **titles** only via that
  fetch; where an item list is too tall for the preview area, it degrades
  cleanly to the same count the column shows (a rendering fallback, not a cut
  feature).
- **`gr list`**: the todo column defaults **off** to keep the default table tight
  (the column-golden-test discipline applies — a new column breaks goldens in
  both `internal/client` and `internal/cli`, and both must be updated with real
  fixture content).
- **GUI**: read + the *human* mutations through `GraithSessionKit` — add the
  todo methods to the `GraithHostClient` actor protocol (the single definition of
  what a session app can do), implement once in `RealHostClient`, and expose a
  todo collection on the shared `FleetModel`/`HostConnection`; each app supplies
  only chrome. The GUI gets add / update / done / block / reopen / remove — but
  **not `claim`**: `todo_claim` is `remoteSessionOnly`, so a remote human in the
  GUI cannot grab an item as a session. That is intended (claiming is a session
  action), and it is why "human mutate" here excludes claim
  ([shared-session-feature-layer](2026-07-14-shared-session-feature-layer.md)).
  The capability is registered in `internal/capabilities/capabilities.json`
  (per-surface `cli`/`ios`/`macos` state) and the doc + GUI fixture regenerated
  (`go test ./internal/capabilities -update`). This is committed Phase 6. A
  full drag-drop *board* renders on the #628 web surface (which does not exist
  yet); the data model here is already board-ready, so it is a Non-Goal by
  external dependency, not by postponement.

#### Scenarios replace `task-done` with the todo system (breaking change)

A scenario scope (`scenario:<id>`) gives the fleet a shared backlog, and this
design **replaces the `task-done` boolean outright** — it does not keep two
parallel notions of "tracked work". `SessionState`/`ScenarioSession.TaskDone`,
`ScenarioSessionInfo.TaskDone` (`messages.go:894`), `ScenarioTaskDone`
(`scenario.go:860`), the `scenario_task_done` control message + handler case,
and `gr scenario task-done` are all **removed**. Breaking backwards
compatibility here is acceptable (per the feature owner); scenario runtime state
is not persisted-as-contract, so no migration shim is needed — the field simply
disappears from the wire type and the manifest is regenerated.

The replacement uses an explicit **assignment**, distinct from claim-ownership:

- A todo item may carry an `assignee` (a scenario member session), set at
  creation or reassigned by the orchestrator. `assignee` is *who is responsible*;
  `owner` is *who is currently working it* (set by the claim CAS). They usually
  coincide but need not — the orchestrator can assign work a member has not yet
  claimed.
- **Seeding.** At scenario start, each member with a non-empty `task`
  (`ScenarioSessionInput.Task`) gets **one seed todo item** in the
  `scenario:<id>` scope, `assignee = that member`, title = the task string. So
  the thing a member used to flip with `gr scenario task-done` is now
  `gr todo done <its-task-item>` — same gesture, real object. A member can add
  sub-items to break its task down.
- **Completion is derived, not declared.** A member is *complete* iff it has ≥1
  assigned item and every assigned item is `done`; a member with **zero**
  assigned items is *"no tracked work"* (shown as `—`, neither pending nor
  complete — this avoids the "flips done immediately" trap a naïve
  owns-nothing-unfinished rule would hit). `gr scenario status` renders
  per-member `done/total` from the assigned items, and the scenario as a whole is
  complete when every member with tracked work is complete.

`gr scenario status`/`list` therefore report progress from **real item state**
instead of a hand-set flag. This is the Phase 7 change; Phases 1–6 already stand
up the store, protocol, scenario-scoped lists, and the `assignee` column that
this builds on.

#### Trade-offs

- **Pro**: purpose-built, indexed, atomically claimable, surfaced, event-driven;
  reuses every existing pattern (a dedicated SQLite store as messaging uses, the
  control-message + CLI + MCP triad, subtree/scenario scoping, `check*` auth).
- **Pro**: the claim CAS is a single statement — provably race-free, and cheap
  to test with concurrent goroutines under `-race`.
- **Con**: a new SQLite DB and the control-message set is real surface area:
  protocol conformance + authmatrix + capability-matrix bookkeeping on every
  addition.
- **Con**: two notions of "tracked work" (todos + `task-done`) coexist through
  Phases 1–6, until Phase 7 *removes* `task-done` and moves scenario progress
  onto assigned todo items — a transitional overlap, not a permanent wart, and a
  deliberate breaking change rather than something punted to a future doc.

### Proposal 2: Typed view over `gr store`

Model todos as JSON/JSONL documents under a reserved store prefix
(`todo/<scope>/<id>.json`), with `gr todo` a thin typed wrapper over
`store.Put`/`List`/`Get`. No new database, no new storage backend, and items are
browsable in an IDE and versioned in git for free.

Rejected as the primary — but the honest reason is *not* that a flat file
"cannot" do CAS. It could: a read/check/write inside the store's existing
exclusive `withLock` (`store.go:179`) would be a correct compare-and-set. The
store simply has **no such public API today** (`Put`/`Get` are the surface, and
a thin `Get`-then-`Put` wrapper genuinely *is* racy), and adding a locked-
transaction API would serialize the *whole store* on every claim and cut a git
commit per status flip — write amplification a live list cannot bear. Layered on
that: cross-item queries ("next unclaimed in this scope", "all blocked")
degrade to directory walks + JSON parsing. So the real case against Proposal 2
is **performance, queryability, and the need to build a specialised locked-
transaction API on the store** — not a theoretical impossibility. Good enough
for a personal scratch list, wrong tool for the fleet workload. (Its
browsability survives as the committed `gr todo export`, Phase 7.)

### Proposal 3: Extend `task-done` into a checklist per session

Grow the scenario session's `TaskDone` boolean (set on the daemon-side
`ScenarioSession`, `scenario.go:890`, reflected onto wire `ScenarioSessionInfo`,
`messages.go:894`) into a list of sub-tasks, keeping everything inside the
scenario state (`state.json`).

Rejected because it is scenario-only (no value for a lone session or an ad-hoc
parent/child pair), it bloats the persisted `state.json` with per-item churn
(the todo write rate does not belong in the session-state file that is
rewritten on every session mutation), and it still has no atomic claim. It is,
in effect, a worse-sited version of what Proposal 1 does properly — which is why
Proposal 1 does not *extend* `task-done` but **replaces** it (Phase 7): scenario
progress moves onto the real todo table (assigned items, derived completion),
and the boolean is deleted rather than grown in place.

## Other Notes

### Answers to the issue's open questions

- **Ownership — who may edit an item?** Create: any in-scope session + human.
  Claim: any in-scope session, with `owner` **server-derived** (never trusted
  from the payload). Transition a claimed item: owner, the override authority
  (subtree anchor root / scenario `OrchestratorID`), or human. The human always
  wins. Enforced for *every* caller via a `checkTodoMutate` helper alongside the
  `auth.go` checks — informed by `checkScenarioOp` (`auth.go:244`) but stricter
  than it (`checkScenarioOp` allows any descendant; the todo rule does not).
- **Relationship to `gr store` — view or subsystem?** A **separate SQLite
  subsystem** (its own `todos.sqlite`, distinct from the jailstore, which is
  merely tables on the msgstore). The store fits documents, not a high-churn
  claimable state machine with indexed queries; the case against it is
  performance/queryability/API, not impossibility (Proposal 2). A read-only
  `gr todo export` to the store is committed (Phase 7) for archival; the table
  stays the source of truth.
- **TUI surfacing — sidebar/picker/overlay?** A `done/total` count column via the
  `sessioncols.go` registry (rides the existing session-metadata path); a
  fleet-wide "Todos" **item** view and a preview panel, which need their own
  `todo_list` fetch + list-model (not a `sessionsForView` filter). `gr list`
  gains an opt-in column.
- **Events — pub/sub on state change?** Yes, opt-in (tri-state `[todo]
  emit_events`): publish compact JSON to `todo:<scope>` via the msgstore
  `graith:system` sender. Subscribed reviewer sessions react without polling, and
  Phase 5 adds the `[trigger.message]` source so **triggers** can react too
  (#592/#593) — committed here, not left as a gap. Best-effort, fail-open,
  `revision`-versioned; the table is authoritative.
- **Overlap with approvals / task-done — replace, extend, or stay separate?**
  **Approvals stay separate — permanently, by design:** approvals
  ([pluggable-approvals-backends](2026-07-03-pluggable-approvals-backends-design.md))
  are a *blocking gate on an action* (an agent asks permission and waits); a
  todo is *non-blocking tracked intent*. Different lifecycles — do not merge.
  **`task-done`, by contrast, is *replaced* by todos (Phase 7, breaking
  change):** the boolean, its command, its control message, and its wire field
  are removed, and scenario progress is derived from assigned todo items (see
  scenario integration). It survives as a boolean only through Phases 1–6 while
  the store/protocol/scenario-list groundwork lands, then it is deleted.

### Implementation Notes — phasing

Shippable increments, each independently testable and coverage-carrying. The
phases are ordered by dependency, **not by whether they ship** — every phase
below is committed work in this design, none is deferred to a later doc. Only
the true Non-Goals (a web kanban board, which depends on the #628 web surface
that does not exist yet; a task-dependency graph, #641; cross-machine lists)
sit outside this plan, and they are excluded because of a real external
dependency or boundary, not postponement.

1. **Phase 1 — store + claim core.** `internal/daemon/todostore.go`: schema with
   `CHECK` constraints, the WAL+`busy_timeout` DSN, `Add`/`List`/`Get`, the
   guarded transition ops, and the atomic `Claim`/`ClaimNext` (with `RETURNING`).
   Includes both reclaim mechanisms — the stop-triggered auto-reopen *and* the
   time-based claim lease (a sweep reopening an `in-progress` item whose
   `updated_at` is older than `[todo] claim_lease`) — plus the `done`-item
   retention sweep (`[todo] retention`). Pure Go + SQLite, no protocol. Full unit
   tests including both concurrent-claim races and both sweeps under `-race`.
   This is the correctness heart — land it first.
2. **Phase 2 — control protocol + CLI.** The request **and response** message
   types (incl. `TodoItemInfo`/`TodoListResponse`), handler cases, authmatrix
   rows, manifest regen, the server-derived `owner`, `checkTodoMutate`, and
   `gr todo` (including `todo_reorder`, whose semantics are settled: reorders are
   serialized through the single daemon handler — one writer — and renumber on
   integer-`position` collision within a transaction, so they are safe by
   construction without a fractional index). Scope anchoring from
   `GRAITH_SESSION_ID`; agent-mode JSON; auto-reopen wired to the session-stop
   path. Integration test through the real daemon (`internal/integration`).
3. **Phase 3 — MCP tools.** The proxy tools (list/add/claim/update/done/block/
   reopen) over the control messages. Lets agents plan/drain durably.
4. **Phase 4 — surfacing.** Overlay count column + a real "Todos" item view
   (its own `todo_list` fetch + list-model) + preview panel; opt-in `gr list`
   column (with golden updates in both packages).
5. **Phase 5 — events + scenario lists + a message-trigger source.**
   `todo:<scope>` pub/sub emission behind the tri-state `[todo] emit_events`;
   `revision`-versioned events; scenario-scoped lists. This phase **also adds the
   third trigger source** — a `[trigger.message]` source
   ([triggers-design](2026-07-11-triggers-design.md)) subscribing to a topic
   glob, with its own binding/restart semantics — so a trigger can react to a
   todo going `blocked`, closing the "compose with triggers" gap directly rather
   than asserting a capability that does not exist. Docs page under
   `website/content/docs/` + `AGENTS.md`.
6. **Phase 6 — GUI.** Todo methods on the `GraithHostClient` actor protocol
   implemented once in `RealHostClient`, a todo collection on
   `FleetModel`/`HostConnection`, and both apps' chrome (a todo list/panel).
   Capability rows in `internal/capabilities/capabilities.json` + regenerated doc
   and GUI fixture. Delivers read + the human mutations (not `claim`) on iOS and
   macOS.
7. **Phase 7 — replace scenario `task-done` + store export (breaking change).**
   *Remove* `TaskDone` from `ScenarioSession`/`ScenarioSessionInfo`, the
   `ScenarioTaskDone` method, the `scenario_task_done` control message + handler
   case + its authmatrix row, and `gr scenario task-done` (regenerate the
   protocol manifest for the dropped wire field). *Add* the `assignee` column,
   seed one assigned todo item per member from its `task` at scenario start
   (`gr scenario start`/`add`), derive per-member `done/total` and scenario
   completion from assigned items, and render them in `gr scenario status`/`list`
   and the overlay's Scenarios view. Also add the read-only `gr todo export
   <scope>` (→ a markdown/JSON doc in the store) for archival. This *retires* the
   coarse primitive rather than leaving two systems.
8. **Phase 8 — auto-mirror from Claude's TodoWrite.** Ingest the agent's in-turn
   checklist into its todo list via the hook stream
   ([claude-hook-events-expansion](2026-07-13-claude-hook-events-expansion.md)),
   so the plan is durable without an explicit `gr todo add`. The noise/churn risk
   is handled by design, not avoidance: mirrored items land in the session's
   **own** anchor (never a shared subtree/scenario list, so one agent's scratch
   plan can't spam a shared backlog), are keyed by a stable content hash so a
   re-emitted TodoWrite updates rather than duplicates, and are tagged
   `source:mirror` so they are visually distinct and separately sweepable.

Fixture strings in tests use Scots words per house style (a `braw` session
draining a `strath` scenario's list, `dreich` items going `blocked`, etc.).

### Testing

- **Claim race** (the headline), two forms: (a) N goroutines claim one explicit
  id — exactly one wins; (b) N goroutines `ClaimNext` over N items — each item id
  is returned exactly once, none twice. Both under `-race`. This is the
  regression that locks the feature's correctness.
- **Transition guards**: `todo_update` cannot change `status`/`scope`/`owner`;
  a status op fails against the wrong pre-state (e.g. `done` on an unclaimed
  item); `owner` on claim is always `ac.sessionID` regardless of any payload
  field.
- **Scope resolution + membership**: a parent and child anchor to the *same*
  `session:<root>` scope and share a list; an unrelated session (ancestor,
  sibling, descendant, unrelated, scenario member, shared-scenario member,
  system orchestrator) is admitted/denied per the predicate table.
- **Authorization**: a sibling cannot transition another agent's claimed item;
  the override authority and human can; a read-only remote guest gets counts,
  never bodies.
- **Reclaim**: when an owner session stops/soft-deletes, its `in-progress` items
  auto-reopen (`owner=''`, `status='todo'`); a still-`done` item is untouched.
- **Integrity**: `CHECK` rejects a bad status and the owner/status invariant; a
  two-level `parent_id` and a cross-scope sub-item are rejected; parent delete
  cascades; `List` is bounded (pagination cap, like the jail list's 2000).
- **Protocol**: `TestManifestUpToDate` / `TestManifestRegistryComplete` and
  `TestRemoteMatrixCompleteness` pass with the new request *and response* types.
- **Events**: a mutation publishes one `todo:<scope>` message with the right
  shape, `System` flag, and monotonic-per-item `revision`; a publish failure is
  non-fatal; emission respects the tri-state `emit_events`.
- **Scenario replacement**: starting a scenario seeds one assigned item per
  member with a `task`; a member with no `task` has no seeded item and reports
  "no tracked work"; marking a member's assigned items `done` makes it (and
  eventually the scenario) complete in `gr scenario status`; the removed
  `scenario_task_done` message is gone from the handler and authmatrix, and
  `TestRemoteMatrixCompleteness` still passes.
- **Persistence**: items and claims survive a daemon restart (WAL DB, unlike the
  runtime-only count field, which rebuilds).
- **CLI/MCP**: golden output for `gr todo list` (both `internal/client` and
  `internal/cli` column goldens); MCP handlers proxy correctly.

### Open questions

Genuine decisions still open for reviewers — *not* deferred plan work (the
plan is fully committed across Phases 1–8; auto-mirror, the claim lease, the
`task-done` replacement, the message-trigger source, the store export and the GUI
are all *in* it). These are design choices within committed phases where the
recommended answer wants a second opinion:

- **Lifecycle edges.** When an anchor root is *purged* (not soft-deleted), does
  its `session:<root>` list archive, orphan, or cascade-delete? When a session is
  *reparented*, does its membership in an ancestor scope change? The design
  leans: a purged anchor's list cascade-deletes with the subtree; scopes follow
  the live tree (membership recomputed, not snapshotted). Worth confirming.
- **Claim-lease default.** Phase 1 builds the lease sweep; the open choice is its
  default value — `[todo] claim_lease` (e.g. 30m, or 0 = stop-based reclaim
  only). Too short reopens work a slow-but-live agent is still doing; too long
  strands a wandered-off worker. A conservative non-zero default is recommended.
- **Retention wiring.** `done` items age out via a Phase 1 sweep; the choice is
  whether it shares `MsgStore.Cleanup`'s *scheduler* while keeping a distinct
  `[todo] retention` age (recommended, so todo and message lifetimes stay
  independent), or reuses `[messages] max_age` outright.
- **Should `gr list`'s todo column be on by default?** Leaning off, to respect
  the "keep the default table tight" convention.
