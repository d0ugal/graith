# Todo List as a First-Class Feature

A shared, persistent, queryable todo list managed by the daemon — a first-class
graith primitive alongside the store, messaging, and scenarios. Items survive
session end, are visible across a session subtree (parent and children) or a
whole scenario, and can be **claimed atomically** so parallel agents pulling
from one list never double-work the same item. Agents drive it through
`gr todo …` and a matching MCP tool; the human sees "what's left" across the
fleet in the overlay and the GUIs.

## Background

An AI agent's plan today lives in its context window. Claude Code renders a
todo checklist while it works, but that list is ephemeral — it is a byproduct
of the model's turn, not a durable object graith can see. When the session
stops, resumes into a fresh context, compacts, or hands off to a sibling, the
plan is gone. This is the same fragility the
[scenarios design](2026-06-22-scenarios.md) calls out: "the setup lives in the
AI's context window and is lost when the session ends."

graith already owns several coordination primitives the agent and the human
share, and each is instructive for how a todo list should be built:

- **The document store** (`internal/store/store.go`,
  [design](2026-06-16-store-refactor-design.md)) — a flat-file, git-backed,
  per-repo key/value store. Operations (`Put`/`Get`/`List`/`Append`/`Remove`)
  are plain filesystem writes serialized by a `store.lock` flock and committed
  to git for history. It is deliberately schema-less: keys are paths, bodies are
  opaque bytes. No daemon connection is needed.
- **Inter-agent messaging** (`internal/daemon/msgstore.go`) — a SQLite-backed
  (`messages.sqlite`, WAL) pub/sub + inbox system. `Publish` inserts a message
  and fans it out to live in-process subscribers (`s.subs[stream]`); `Read`,
  `Ack`, and cursors track per-subscriber unread state. The daemon publishes
  its own automated notices under a `graith:system` sender (`isSystemSender`).
- **Scenarios** (`internal/daemon/scenario.go`,
  [design](2026-06-22-scenarios.md)) — declarative multi-session fleets. Each
  scenario session carries a `role`/`task`, and `gr scenario task-done` flips a
  single `TaskDone bool` on `ScenarioSessionInfo` (`ScenarioTaskDone`,
  `scenario.go:860`). That is the closest thing graith has to "tracked work"
  today — one boolean per session, no sub-items, no ordering, no claiming.
- **The session tree** — every `SessionState` has a `ParentID`
  (`state.go:56`) and an optional `ScenarioID` (`scenario.go`). `isDescendantOf`
  (`auth.go:335`) walks the parent chain, and this is how hierarchical
  operations (`gr stop --children`, `gr msg send --parent`) and scenario
  authorization already scope themselves.

The wire protocol is a `{"type":…,"payload":…}` envelope on channel `0x00`
(`protocol/messages.go`, dispatched in `daemon/handler.go`). Every control
message is classified in `remoteMessagePolicy` (`daemon/authmatrix.go`) — a
missing entry fails closed and trips `TestRemoteMatrixCompleteness` — and every
wire struct is reflected into the Go↔Swift conformance manifest
(`protocol/manifest.go`,
[protocol-conformance-design](2026-07-14-protocol-conformance-design.md)).

Agents also reach the daemon over MCP: `internal/mcp/server.go` registers tools
with `gomcp.AddTool` (`list_sessions`, `create_session`, `publish_message`,
`read_inbox`, …). Each handler is a thin proxy — it dials the daemon socket via
`s.connect()` and sends the same control message the CLI would
(`createSession`, `server.go:255`). So a CLI verb and an MCP tool are two front
doors onto one daemon operation.

The GUIs (iOS + macOS) bind to a shared Swift layer, `GraithSessionKit`
([shared-session-feature-layer](2026-07-14-shared-session-feature-layer.md)),
and every frontend capability is tracked in the hand-maintained matrix at
`internal/capabilities/capabilities.json`, guarded against doc- and code-drift.

## Problem

1. **Plans do not survive the context window.** A stopped, resumed, compacted,
   or handed-off agent loses its plan. There is no durable record of "what I was
   going to do" that graith — or a successor agent — can read back.

2. **No shared view across a subtree or fleet.** A parent orchestrator and its
   children, or the sessions in a scenario, have no common list of outstanding
   work. Coordination is improvised over free-text messages, which are a log,
   not a queryable task set.

3. **No safe hand-out for parallel agents.** When an orchestrator wants three
   agents to drain a backlog, there is no collision-free way to hand out items.
   Two agents reading the same list will both "start" the same task — a
   read-then-write TOCTOU race. The issue's competitive analysis
   ([amux/mixpeek](https://github.com/d0ugal/graith/issues/591#issuecomment-4840251519))
   identifies **atomic claim** as the load-bearing correctness primitive.

4. **The human has no "what's left" at a glance.** `gr list` shows session
   status, not the work each session is carrying. There is no fleet-wide answer
   to "how much is outstanding, and where is it blocked?"

5. **`task-done` is too coarse.** A scenario session's work is a single
   boolean. It cannot express "3 of 7 items done, 1 blocked" — the granularity
   an orchestrator actually needs to reason about progress.

## Goals

- A **durable** todo list owned by the daemon, surviving session end, resume,
  compaction, and daemon restart.
- **Item shape** rich enough to be useful: id, title, status
  (`todo`/`in-progress`/`done`/`blocked`), owner, optional parent item, tags,
  timestamps.
- **Scoping** to a session subtree by default, with the option to scope to a
  scenario — matching how graith already draws coordination boundaries.
- **Atomic, collision-free claim**: a compare-and-set `todo → in-progress`
  bound to a session, so parallel agents never double-claim (issue's
  load-bearing requirement).
- **Two front doors**: `gr todo …` CLI and a matching MCP tool, so agents
  manage items directly and the human drives from the shell.
- **Surfacing**: a fleet-wide and per-session view in the overlay, and a
  read (at least) view in the GUIs via `GraithSessionKit`.
- **Optional events**: state changes can emit pub/sub so triggers (#592, #593)
  and reviewers can react.
- Clear-eyed on **overlap**: complement, not replace, `task-done` and
  approvals.

### Non-Goals

- **A visual kanban / drag-drop board.** The issue floats a vibe-kanban-style
  board ([comment](https://github.com/d0ugal/graith/issues/591#issuecomment-4842556611));
  that is a GUI presentation concern layered on top of this data model and
  belongs to the web-surface work (#628), not v1.
- **A full task-dependency graph.** Sub-items (one level of parent/child) are in
  scope; arbitrary DAG dependencies and topological scheduling are #641's
  problem, not this doc's.
- **Cross-machine todo lists.** Like scenarios, todos are single-daemon.
- **Replacing approvals or `task-done`.** See the overlap section — they stay.
- **Rich text / attachments / comments on items.** Titles + tags + an optional
  one-line note. Discussion happens in messaging, which items can reference.
- **Auto-population from the agent's own in-turn checklist.** Mirroring Claude's
  live TodoWrite into graith is an attractive follow-up (see Open questions) but
  is not required for v1.

## Proposals

### Proposal 0: Do Nothing

Agents keep planning in-context; scenarios keep using the one-bit `task-done`;
orchestrators keep improvising over messages and the store.

This is the status quo, and it is genuinely insufficient for the reasons in the
Problem section: no durability, no shared subtree/fleet view, and — most
sharply — **no collision-free hand-out**, which makes graith's headline
multi-agent story (scenarios, fan-out) unsafe the moment two agents pull from
one backlog. A motivated user *can* approximate a todo list today by
`gr store append`-ing JSONL lines, but the store has no atomic CAS (it is a
whole-file flock, not a row-level compare-and-set), no schema, no query, and no
surfacing. Rejected.

### Proposal 1: Daemon-owned todo subsystem with atomic claim (Recommended)

A purpose-built todo subsystem in the daemon, backed by SQLite, exposed through
control messages, a `gr todo` CLI, and an MCP tool, scoped to a session subtree
or a scenario, with an atomic compare-and-set claim as its correctness
centrepiece.

#### Data model

One `todos` table in a new SQLite database, `todos.sqlite`, in the data dir
(added to `config.Paths` next to `MessagesDB`, `paths.go:88`). SQLite — not the
flat-file store — because the claim semantics **require** a single-statement
atomic compare-and-set, which a git-backed flat file cannot give us
(`store.lock` serializes whole operations but there is no row-level CAS, and a
git commit per state change would be absurdly heavy). We already run SQLite for
messaging, so this adds no dependency.

```go
// internal/daemon/todostore.go
type TodoItem struct {
    ID        string   `json:"id"`         // "td-" + random hex
    Title     string   `json:"title"`
    Status    string   `json:"status"`     // todo | in-progress | done | blocked
    Scope     string   `json:"scope"`      // "session:<id>" | "scenario:<id>"
    Owner     string   `json:"owner,omitempty"`   // session ID that claimed it
    ParentID  string   `json:"parent_id,omitempty"` // one-level sub-item
    Tags      []string `json:"tags,omitempty"`
    Note      string   `json:"note,omitempty"`      // optional one-liner
    CreatedBy string   `json:"created_by"`          // session ID / "human"
    CreatedAt string   `json:"created_at"`
    UpdatedAt string   `json:"updated_at"`
    Position  int64    `json:"position"`            // manual ordering within a scope
}
```

Schema (mirroring the `initSchema` style in `msgstore.go:67`), with `tags`
stored as a joined string or a child `todo_tags` table (join table preferred so
tag filters are indexable):

```sql
CREATE TABLE IF NOT EXISTS todos (
    id          TEXT PRIMARY KEY,
    title       TEXT NOT NULL,
    status      TEXT NOT NULL DEFAULT 'todo',
    scope       TEXT NOT NULL,
    owner       TEXT NOT NULL DEFAULT '',
    parent_id   TEXT,
    note        TEXT NOT NULL DEFAULT '',
    created_by  TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL,
    position    INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_todos_scope       ON todos(scope, status, position);
CREATE INDEX IF NOT EXISTS idx_todos_owner       ON todos(owner) WHERE owner <> '';
CREATE INDEX IF NOT EXISTS idx_todos_parent      ON todos(parent_id) WHERE parent_id IS NOT NULL;
```

**Scope is the primary key of "which list."** `scope = "session:<id>"` means the
item belongs to a session subtree; `scope = "scenario:<id>"` means a scenario's
shared list. There is no free-floating global list — every item is anchored, the
way every session has a `ParentID`/`ScenarioID`.

#### The claim: atomic compare-and-set

The load-bearing detail from the issue. Claiming is a single conditional
`UPDATE` whose `WHERE` guards the pre-state, exactly the pattern the jailstore
already uses for idempotent release (`jailstore.go:178`,
`UPDATE … WHERE id = ? AND released_at = ''`):

```go
func (s *TodoStore) Claim(id, owner string) (bool, error) {
    res, err := s.db.Exec(
        `UPDATE todos SET status='in-progress', owner=?, updated_at=?
         WHERE id=? AND status='todo' AND owner=''`,
        owner, now, id)
    if err != nil { return false, err }
    n, _ := res.RowsAffected()
    return n == 1, nil   // exactly one claimant wins; losers get n==0
}
```

Two agents racing to claim the same item: SQLite serializes the writes, the
first `UPDATE` matches and sets `owner`, the second matches zero rows. The
loser is told "already claimed" and moves on. No TOCTOU window, no read-then-
write. A `ClaimNext(scope, owner)` convenience does the same against "the
lowest-`position` `todo` item in this scope" in one statement, so a fan-out
agent can pull work with a single call.

#### Ownership and mutation rules

Answering the issue's open question directly:

- **Create**: any session may add items to **its own scope** (its subtree) or a
  scenario it belongs to. The human (local CLI) may add to any scope.
- **Claim / start / complete / block**: any session **in the item's scope** may
  claim an unclaimed item; only the **owner**, the **subtree/scenario
  orchestrator**, or the **human** may transition an item that is already
  claimed. This mirrors `checkTarget`/`isDescendantOf` (`auth.go`) and the
  scenario-management rule (`checkScenarioManage`, `auth.go:240`): the
  orchestrator is the control plane and may override; a plain sibling may not
  steal another agent's in-progress item.
- **Edit title/tags/reorder/delete**: owner, orchestrator, or human.
- **The human always wins** (`roleLocalHuman` / paired human), consistent with
  every other subsystem.

Enforcement lives in the handler using the existing `authContext` and a new
`checkTodoMutate(ac, item)` helper next to the other `check*` functions in
`auth.go`, so the todo subsystem inherits the fail-closed local-auth model
unchanged.

#### Storage: separate subsystem, not a store view

Answering the second open question: **a separate subsystem**, not a typed view
over `gr store`. The store is the wrong substrate for three concrete reasons —
(1) no row-level atomic CAS (the whole point), (2) a git commit per status flip
is far too heavy for the write rate a live todo list sees, and (3) queries
("all blocked items in this scenario", "next unclaimed") want an indexed table,
not a directory walk. The todo subsystem is its own SQLite store, sibling to the
msgstore, exactly as the jailstore is. (A read-only `gr store`-style export —
dumping a scope to a markdown/JSON doc for archival — is a cheap, nice follow-up,
but the source of truth is the table.)

#### Control protocol

New message types in `protocol/messages.go`, each registered in `registeredTypes`
+ classified in `swiftAnnotations` and the manifest regenerated
(`go test ./internal/protocol -run TestManifestUpToDate -update`), each with a
`remoteMessagePolicy` row (`authmatrix.go`) or `TestRemoteMatrixCompleteness`
fails:

| Message | Payload | Policy | Notes |
|---|---|---|---|
| `todo_add` | scope, title, tags, parent_id, note | `remoteHumanRW` | create |
| `todo_list` | scope, filter (status/tag/owner) | `remoteReadOnly` | query |
| `todo_claim` | id (or scope for claim-next), owner | `remoteSessionOnly` | atomic CAS |
| `todo_update` | id, status/title/tags/note | `remoteHumanRW` | handler re-checks owner/orch |
| `todo_reorder` | id, position | `remoteHumanRW` | |
| `todo_remove` | id | `remoteHumanRW` | |

`todo_claim` is `remoteSessionOnly` — a session acting on itself, like
`approval_request`/`status_report` (`authmatrix.go:60`) — so a paired human
can't impersonate a session to grab an item. Reads are `remoteReadOnly` so the
GUI/guest can observe. Handler cases go in `daemon/handler.go`'s dispatch loop,
each delegating to a `SessionManager` method that calls the `TodoStore` under
the daemon lock where scope resolution needs the session tree.

#### CLI surface

`internal/cli/todo.go`, one file, mirroring `cli/scenario.go`'s shape:

```bash
gr todo add "Wire the claim CAS" --tag backend --tag p1   # add to my scope
gr todo add "Sub-task" --parent td-abc123                 # one-level sub-item
gr todo list                          # my subtree's items, grouped by status
gr todo list --scenario tracing       # a scenario's shared list
gr todo list --status blocked --all   # fleet-wide, filtered (human)
gr todo claim td-abc123               # atomic claim → in-progress, owned by me
gr todo next                          # claim the next unclaimed item in my scope
gr todo start td-abc123               # alias: claim
gr todo done td-abc123                # → done
gr todo block td-abc123 "waiting on API review"  # → blocked + note
gr todo reopen td-abc123              # → todo, clears owner
gr todo rm td-abc123
```

Scope auto-resolves from `GRAITH_SESSION_ID` (an agent operates on its own
subtree) exactly as `gr store`/`gr status` resolve today; `--scenario <name>`
and `--session <id>` override; `--all` (human/orchestrator) spans scopes. Agent
mode auto-enables `--json` (the existing `GR_AGENT_MODE` detection), so an agent
gets structured output for free.

#### MCP tool surface

New tools in `internal/mcp/server.go` via `gomcp.AddTool`, each a thin proxy
over the control messages above (the `createSession` pattern, `server.go:255`):

- `todo_list(scope?, status?, tag?)` → items
- `todo_add(title, tags?, parent_id?, note?)` → item
- `todo_claim(id?)` → `{claimed: bool, item}` (id omitted = claim next)
- `todo_update(id, status?, title?, tags?, note?)` → item

An agent can then plan durably in one call and drain a shared backlog with
`todo_claim` — the fan-out story the issue is really about.

#### Events (optional, on by default for scenario scopes)

Answering the events open question: **yes, opt-in pub/sub on state change**,
reusing the msgstore. On every mutation the daemon publishes a compact JSON
event to a well-known topic — `todo:<scope>` — via `MsgStore.Publish` with the
`graith:system` sender (so it is flagged `System`, `msgstore.go:219`):

```json
{"event":"claimed","id":"td-abc","scope":"scenario:sc-1","owner":"def456","status":"in-progress"}
```

This composes for free with triggers: a `[trigger.watch]`-style consumer, or a
reviewer session subscribed to `todo:scenario:sc-1`, reacts to "an item was
marked `done`" or "an item went `blocked`" without polling. It is a pure
accelerator — the table is the source of truth — so it is fail-open and gated
by a `[todo] emit_events` config flag (default on for scenario scopes, off for
lone-session scopes to avoid noise).

Note the distinction the notify machinery already draws (`daemon/notify.go`): a
bare `MsgStore.Publish` to a topic reaches *subscribers and the GUI* but does
**not** wake a stopped agent — only `notifyFromDaemon` (`notify.go:79`) routes an
event into `inbox:<id>` and auto-resumes the session. So a todo event uses plain
`Publish` by default (observation), and *escalates* to `notifyFromDaemon` only
for the cases worth interrupting an agent over — e.g. "an item you own was
`blocked`" delivered to that owner's inbox, matching how PR/CI notices already
wake their owning session.

#### Surfacing in the TUI and GUI

- **The column registry does the heavy lifting.** Both `gr list` and the overlay
  picker draw trailing columns from one source of truth,
  `internal/client/sessioncols.go` (`SessionColumn`, with `ShowCLI`/`ShowTUI`
  flags and paired `CLIValue`/`TUIValue` formatters). Adding a `done/total` todo
  column there makes it appear in *both* surfaces at once — the overlay delegate
  iterates `tuiColumns()` and the picker auto-grows its width
  (`overlay.go:517`/`:681`). The count reaches the client via the established
  derive-and-publish path
  ([sidebar-metadata-design](2026-06-25-sidebar-metadata-design.md)): a runtime
  `json:"-"` field on `SessionState`, copied in `toSessionInfo` onto
  `protocol.SessionInfo` (`omitempty`), so CLI, TUI, and GUI all read one wire
  type with no per-surface side channel.
- **Overlay preview + view mode**: a todo panel appended to the per-session
  metadata block in the preview pane (`overlay.go:2226-2321`), and a fleet-wide
  **"Todos"** view mode added to the existing enum (`viewAll`,
  `viewNeedsAttention`, `viewActive`, `viewStarred`, `viewScenario`,
  `viewDeleted` — `overlay.go:34`) plus a filter in `sessionsForView()`, listing
  outstanding items across scopes: the human's "what's left".
- **`gr list`**: the todo column defaults **off** to keep the default table tight
  (the column-golden-test discipline applies — a new column breaks goldens in
  both `internal/client` and `internal/cli`, and both must be updated with real
  fixture content).
- **GUI**: read + basic mutate through `GraithSessionKit` — add the todo methods
  to the `GraithHostClient` actor protocol (the single definition of what a
  session app can do), implement once in `RealHostClient`, and expose a todo
  collection on the shared `FleetModel`/`HostConnection`; each app supplies only
  chrome
  ([shared-session-feature-layer](2026-07-14-shared-session-feature-layer.md)).
  The capability is registered in `internal/capabilities/capabilities.json`
  (per-surface `cli`/`ios`/`macos` state) and the doc + GUI fixture regenerated
  (`go test ./internal/capabilities -update`). A full drag-drop board is a
  later, web-surface (#628) concern (Non-Goal).

#### Integration with scenarios and `task-done`

A scenario scope (`scenario:<id>`) gives the fleet a shared backlog. The natural
evolution of `task-done`: instead of one boolean per session, a scenario session
completes when its owned items are all `done`. v1 keeps `task-done` as-is (see
overlap section) and simply lets a scenario carry a todo list; a follow-up can
derive `TaskDone` from "this session owns no unfinished items."

#### Trade-offs

- **Pro**: purpose-built, indexed, atomically claimable, surfaced, event-driven;
  reuses every existing pattern (SQLite subsystem like msgstore/jailstore,
  control-message + CLI + MCP triad, subtree/scenario scoping, `check*` auth).
- **Pro**: the claim CAS is a single statement — provably race-free, and cheap
  to test with concurrent goroutines under `-race`.
- **Con**: a new SQLite table and ~6 control messages is real surface area:
  protocol conformance + authmatrix + capability-matrix bookkeeping on every
  addition.
- **Con**: two sources of "tracked work" (todos + `task-done`) until a later doc
  unifies them — a transitional wart called out honestly below.

### Proposal 2: Typed view over `gr store`

Model todos as JSON/JSONL documents under a reserved store prefix
(`todo/<scope>/<id>.json`), with `gr todo` a thin typed wrapper over
`store.Put`/`List`/`Get`. No new database, no new storage backend, and items are
browsable in an IDE and versioned in git for free.

Rejected as the primary because it cannot deliver the **atomic claim** — the
store's `withLock` flock serializes whole file operations but offers no
row-level compare-and-set, so two agents can still both read "unclaimed", both
write "mine", and the last writer wins silently (exactly the TOCTOU the issue
forbids). A git commit per status transition is also far too heavy for a live
list's write rate, and cross-item queries degrade to directory walks + JSON
parsing. Good enough for a personal scratch list, unsafe for the fleet use case
that motivates the feature. (Its browsability survives as the optional
store-export follow-up in Proposal 1.)

### Proposal 3: Extend `task-done` into a checklist per session

Grow `ScenarioSessionInfo.TaskDone` (`messages.go:894`) from a boolean into a
list of sub-tasks, keeping everything inside the scenario state
(`state.json`).

Rejected because it is scenario-only (no value for a lone session or an ad-hoc
parent/child pair), it bloats the persisted `state.json` with per-item churn
(the todo write rate does not belong in the session-state file that is
rewritten on every session mutation), and it still has no atomic claim. It does,
however, point at the right *end state* — todos and `task-done` converging — and
Proposal 1 leaves room for that convergence rather than fighting it.

## Other Notes

### Answers to the issue's open questions

- **Ownership — who may edit an item?** Create: any session in-scope + human.
  Claim: any in-scope session (atomic CAS). Mutate a claimed item: owner,
  subtree/scenario orchestrator, or human. The human always wins. Enforced via
  a `checkTodoMutate` helper alongside the existing `auth.go` checks.
- **Relationship to `gr store` — view or subsystem?** A **separate SQLite
  subsystem** (sibling to msgstore/jailstore). The store cannot provide atomic
  CAS, cheap status churn, or indexed queries. An optional read-only store
  export is a follow-up, not the source of truth.
- **TUI surfacing — sidebar/picker/overlay?** All three, minimally: a per-session
  count badge in the picker, a todo panel in the overlay preview, and a
  fleet-wide "Todos" view mode. `gr list` gains an opt-in `done/total` column.
- **Events — pub/sub on state change?** Yes, opt-in: publish compact JSON to
  `todo:<scope>` via the msgstore `graith:system` sender, so triggers (#592,
  #593) and reviewer sessions react without polling. Fail-open accelerator; the
  table is authoritative.
- **Overlap with approvals / task-done — replace, extend, or stay separate?**
  **Stay separate in v1**, with a convergence path. Approvals
  ([pluggable-approvals-backends](2026-07-03-pluggable-approvals-backends-design.md))
  are a *blocking gate on an action* (an agent asks permission and waits); a
  todo is *non-blocking tracked intent* (a plan item nobody is blocked on).
  Different lifecycles — do not merge. `task-done` is the one to eventually
  subsume: a scenario todo list is a strict superset of one boolean, and a later
  doc can derive `TaskDone` from "owns no unfinished items." Until then they
  coexist.

### Implementation Notes — phasing

Shippable increments, each independently testable and coverage-carrying:

1. **Phase 1 — store + claim core.** `internal/daemon/todostore.go`: schema,
   `Add`/`List`/`Get`/`Update`/`Remove`, and the atomic `Claim`/`ClaimNext`.
   Pure Go + SQLite, no protocol. Full unit tests including the concurrent-claim
   race under `-race` (spawn N goroutines claiming one item; assert exactly one
   `RowsAffected() == 1`). This is the correctness heart — land it first.
2. **Phase 2 — control protocol + CLI.** The six message types, handler cases,
   authmatrix rows, manifest regen, `checkTodoMutate`, and `gr todo`. Scope
   auto-resolution from `GRAITH_SESSION_ID`; agent-mode JSON. Integration test
   through the real daemon (`internal/integration`).
3. **Phase 3 — MCP tools.** Four tools proxying the control messages. Lets
   agents plan/drain durably.
4. **Phase 4 — surfacing.** Overlay count badge + preview panel + "Todos" view
   mode; opt-in `gr list` column (with golden updates in both packages).
5. **Phase 5 — events + scenario integration.** `todo:<scope>` pub/sub emission
   behind `[todo] emit_events`; scenario-scoped lists; the docs page under
   `website/content/docs/` and `AGENTS.md`.
6. **Phase 6 (deferred) — GUI + convergence.** `GraithSessionKit` todo model +
   capability-matrix wiring; later, derive scenario `task-done` from todo
   completion; optional store export.

Fixture strings in tests use Scots words per house style (a `braw` session
draining a `strath` scenario's list, `dreich` items going `blocked`, etc.).

### Testing

- **Claim race** (the headline): concurrent goroutines claim one item;
  exactly one succeeds. Under `-race`. This is the regression test that locks
  the whole feature's correctness.
- **Scope isolation**: an item in `session:A` is invisible to `session:B` (not a
  descendant); a scenario item is visible to all its members.
- **Authorization**: a sibling cannot mutate another agent's claimed item; the
  orchestrator and human can; a soft-deleted session's scope is inert.
- **Protocol**: `TestManifestUpToDate` / `TestManifestRegistryComplete` and
  `TestRemoteMatrixCompleteness` pass with the new messages.
- **Events**: a mutation publishes exactly one `todo:<scope>` message with the
  right shape and `System` flag; disabled cleanly when `emit_events=false`.
- **Persistence**: items survive a daemon restart (WAL DB, unlike runtime-only
  `SessionState` fields); a claim survives too.
- **CLI/MCP**: golden output for `gr todo list`; MCP handlers proxy correctly.

### Open questions

- **Auto-mirror from Claude's TodoWrite?** Claude Code emits a live in-turn
  checklist. Should a hook ingest it into the daemon todo list automatically, so
  the agent's own plan becomes durable without an explicit `gr todo add`? Very
  attractive, but risks noise and duplicate-item churn; deferred, and it depends
  on the hook-events expansion
  ([claude-hook-events-expansion](2026-07-13-claude-hook-events-expansion.md)).
- **Retention.** Do `done` items live forever, or age out like messages
  (`[messages] max_age`)? Leaning toward a `[todo] retention` sweep of `done`
  items, keeping the table small.
- **Position/reordering semantics** for concurrent reorders — a simple integer
  `position` with "gap" allocation, or a fractional index? Start with integers +
  a renumber-on-collision; revisit if it churns.
- **Should `gr list`'s todo column be on by default?** Leaning off, to respect
  the "keep the default table tight" convention.
